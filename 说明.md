# 广东鸿轩实业集团展示中心（阳江工厂）—— 项目文件架构说明

> **设计方**: 筑造实践科技  
> **项目周期**: 2023年12月 ~ 2024年10月（约 11 个月）  
> **核心任务**: 阳江工厂新办公楼建筑幕墙 + 室内外一体化装修工程设计  
> **建筑概况**: 地上 4 层（1F~3F + 屋面层），含前厅接待区、大会议室、展厅、茶室、办公室等功能空间  
> **文件总量**: 约 2,690 个文件 | 总大小约 4.5 GB+  
>
> **文件位置**: "E:\筑造实践\阳江工厂"
>
> **设计工具链**: Rhino 3D → Enscape 渲染 → AutoCAD / 天正 施工图 → Excel 物料管理

> ⓘ **关于「天正」**：天正（Tangent）是国内建筑行业广泛使用的 AutoCAD 二次开发平台，提供面向建筑、结构、给排水、电气等专业的专用绘图工具。工作空间中 `_t3` 后缀的文件即为天正格式（TArch 版本），文件体积通常比普通 .dwg 更大，因为携带了天正自定义实体数据。

## 二、三大 AI Agent 展区

本项目文件架构可天然支撑三个 AI Agent，以下分别说明每个 Agent 的数据基础、业务逻辑与对应的文件资源。

---

### Agent 1：BOM 智能审核 Agent（数据最硬）

**业务场景**: 建筑装修工程中，物料清单（BOM）品类多、数量大、版本迭代频繁，人工审核容易出错。该 Agent 自动比对不同版本的物料清单，检测数量异常、品类遗漏、单价波动，并生成审核报告。

#### 数据基础

| 数据类型 | 核心文件 | 数据量 |
|----------|----------|--------|
| **工程统量** | `EXCEL/20240830_广东鸿轩实业集团展示中心工程统量.xlsx` | 265 MB（超大） |
| **装饰材料下料清单** | `EXCEL/装饰材料下料清单2024-09-6.xlsx` | 84 MB |
| **吊顶材料清单** | `EXCEL/阳江吊顶0915.xlsx` | 59 MB |
| **物料清单** | `EXCEL/阳江物料清单20240428.xlsx` | 41 MB |
| **室内洁具物料清单** | `EXCEL/室内洁具物料清单20240601.xlsx` | 6.9 MB |

| **灯光清单** | `IN/灯光厂家/阳江灯光0831.xlsx` / `0902.xlsx` | — |

#### Agent 核心能力

1. **跨版本差异比对**: 自动识别同一物料清单在不同版本间的数量变化、新增/删除条目
2. **品类完整性检查**: 对照标准物料分类体系（墙板、框架、门窗、洁具、电器等），检测遗漏品类
3. **数量逻辑校验**: 结合 CAD 图纸中的面积/尺寸数据，反向校验 BOM 数量的合理性
4. **供应商报价审核**: 比对报价单与预算控制目标，标出异常偏高/偏低项
5. **审图联动**: 当 BOM 发生变更时，自动关联对应的 CAD 加工图版本

#### 用到的最火 AI 应用

| AI 技术 | 在 Agent 中的角色 | 为什么选它 |
|---------|------------------|-----------|
| **Code Interpreter**<br>（OpenAI / Claude 原生支持） | 直接在沙箱中运行 Python，读取 17 个 Excel 的多 sheet，执行交叉校验、聚合比对 | 无需预建数据库或 ETL 管线，零基础设施门槛；分析师可直接用自然语言描述校验规则 |
| **Structured Output**<br>（Function Calling / JSON Mode） | 将审核结果以标准化 JSON schema 输出（物料编号、问题类型、严重等级、建议操作） | 前端可直接渲染为审核看板，无需后处理解析；同时支持下游系统（钉钉/企微消息推送）消费 |
| **Agentic Workflow**<br>（Multi-node State Machine） | 多节点流程：解析 Excel → 交叉校验 → 异常标记 → 优先级排序 → 生成审核报告 | 每个节点可独立调试、独立回退；审核灰度发布时可按节点开关，A/B 测试某个校验规则的准确率 |

**技术亮点举例**：门表中有"30 扇门、70.66㎡"，物料清单中门类条目合计却只有 28 扇——Code Interpreter 在 3 秒内完成 6 个 sheet 的跨表关联查询，标出这 2 扇门的缺口，Structured Output 返回一条 JSON：`{"issue_type": "quantity_mismatch", "source": "门表", "target": "物料清单", "delta": -2, "severity": "high"}`。

#### 亮点数据

- 墙板信息表有 **4 个版本**，跨越 2024-06-21 → 2024-07-10
- 装修工程清单有 **6 个版本**，跨越 2024-08-18 → 2024-09-24
- 物料管理文件总量超过 **500 MB**，包含数千条物料记录

**业务场景**: 工程实践中，图纸的迭代频率和密度是最真实的进度压力指标。该 Agent 通过追踪关键图纸的修改频率、版本间隔、迭代密度，结合现场施工照片的时间戳，识别进度滞后风险，提前发出预警。

#### Agent 核心能力

1. **图纸迭代密度监控**: 计算每类图纸的日均修改次数，超过阈值自动预警
2. **版本间隔分析**: 检测连续两次修改间隔是否异常（过密=返工压力，过疏=停滞风险）
3. **现场-图纸一致性校验**: 比照现场照片时间戳与图纸版本日期，判断施工是否按图执行
4. **多专业耦合延迟检测**: 当结构/幕墙/水电图纸迭代不同步时，发出接口协调风险警告
5. **进度偏差自动计算**: 对照片划进度表，输出偏差天数、受影响范围

#### 用到的最火 AI 应用

| AI 技术 | 在 Agent 中的角色 | 为什么选它 |
|---------|------------------|-----------|
| **RAG（检索增强生成）** | 将会议纪要（3 月 13 日会议 14 条决策）、业主指令、现场勘测记录向量化存入知识库；Agent 分析每个任务的依赖关系时，自动检索关联的决策上下文 | 进度表只有"负责""分配""截止"字段，缺少"为什么延期""依赖谁"的语义信息——RAG 补齐了这块缺失的因果链条 |
| **多模态理解（Vision）** | 上传现场施工照片（1,651 张 HEIC/JPG），与对应日期的 Enscape 效果图做视觉对比，辅助判断实际施工进度是否与设计预期一致 | 效果图 vs 现场照片的视觉差异比文字描述更直观；Vision 模式可识别"墙板未安装""吊顶未完成"等结构化进度状态 |
| **Structured Output + 甘特图渲染** | 输出标准 JSON（任务 ID、风险等级、影响链、建议操作），前端用 ECharts 渲染风险热力甘特图 | 产品经理可以直接在周会上投放甘特图，红色热力区直观标示哪些任务处于"高密度迭代→可能延期"区间 |

**技术亮点举例**：进度表显示"框架加工图"任务截止日期为 8 月 15 日，但 RAG 检索到 8 月初的幕墙变更记录——幕墙图在 8 月 9 日才更新完，导致框架加工图的依赖项延后了 6 天。Agent 自动将这条任务标为红色，同时用 Vision 模式比对 8 月 20 日的现场照片，确认框架尚未安装完毕，验证了预警的准确性。

#### 叙事亮点

安装示意图在 **55 天内迭代 10 版** → 平均每 5.5 天一版；墙板模型在 **36 天内迭代 7 版** → 平均每 5 天一版。这种高密度迭代充分展示了工程后期的设计-施工协同压力，是进度风险预警的最佳素材。

### Agent 3：设计变更追踪 Agent（最体现产品思维）

**业务场景**: 建筑设计过程中，变更是常态，但变更的溯源、影响范围评估、多专业同步更新是巨大痛点。该 Agent 自动追踪每次设计变更的触发原因、影响范围、关联文件，生成变更影响报告，并提醒受影响专业。

#### 数据基础

**（1）有明确迭代链的图纸 —— 变更轨迹**

```
结构施工图:
  2024-03-04 (初版) → 03-18 (加地坪范围) → 04-28 → 05-10 → 05-11 → 05-24 (终版)
  影响下游: 幕墙支座反力计算（05-02）、框架加工图（06-20起）

安装示意图:
  0731 → 0803 → 0809 → 0812 → 0817 → 0827 → 0828 → 0830 → 0901 → 0905 → 0922
  影响下游: 墙板加工图、转角板加工图、连接件加工图

新缝设计:
  0803 → 0812 → 0817 → 0814(t3) → 0901 → 0905
  影响下游: 全部墙板模型、框架加工图

墙板模型:
  0726 → 0804 → 0810 → 0814 → 0819 → 0827 → 0826/0831 (终版)
  影响下游: 墙板信息表、转角板标签、单元板表
```

#### Agent 核心能力

1. **变更溯源**: 自动关联会议纪要、现场勘测、业主指令到具体的图纸变更版本
2. **影响范围分析**: 当某个专业图纸发生变更时，自动识别下游受影响的所有文件和专业
3. **变更通知推送**: 按专业角色自动生成变更通知单，标注受影响范围和更新要求
4. **变更履历可视化**: 以时间轴形式展示每个设计元素的完整变更历史
5. **未同步检测**: 检测是否存在"上游已变更但下游未更新"的断链情况

#### 用到的最火 AI 应用

| AI 技术 | 在 Agent 中的角色 | 为什么选它 |
|---------|------------------|-----------|
| **AI Code Interpreter** | 读取同一张表的多个历史版本（如门表 20240309 vs 20240505），用 Python 做结构化 Diff：哪些门增减了、尺寸变了、材料换了 | Excel 版本比对不是简单的 cell-by-cell 比较——Code Interpreter 可以理解"这是同一扇门、改了尺寸"与"这是新增的一扇门"的语义差异 |
| **Document AI（PDF/Word 解析）** | 解析会议纪要 Word/PDF（3 月 13 日），自动提取 14 条决策关键词（"取消太阳能""增加烟感""旧楼外立面全封闭"），建立决策-变更关联索引 | 会议纪要是非结构化文本，Document AI 将"决策"从自然语言中结构化为可查询的事件节点，与图纸变更时间轴对齐 |
| **Agentic Loop + Self-Reflection** | Agent 生成变更报告后，进入自评循环：逐条检查是否遗漏了影响范围、是否所有下游专业都已通知、报告中是否有逻辑矛盾 | 变更追踪是最容易"漏球"的场景——Self-Reflection 让 Agent 像审图人员一样二次核查自己的输出，大幅降低误报和漏报 |

**技术亮点举例**：门表从 3 月 9 日到 5 月 5 日经历了一次静默变更——3 扇防火门尺寸从 900×2100 改为 1000×2200。没有会议纪要直接提及这个改动，但 Document AI 从 3 月 13 日会议纪要中提取到"消防规范升级"的决策关键词，Agentic Loop 在第二轮自评时发现了这条关联，补上了变更原因的拼图。整个链条：消防规范升级 → 防火门尺寸变更 → 洞口收边件加工图需更新 → 1 人自动通知。

#### 产品思维体现

这个 Agent 不是简单的版本比较工具，而是模拟了一个"设计经理"的工作流：理解变更原因 → 评估影响范围 → 通知相关人员 → 追踪执行闭环。它体现了产品经理对业务流程的深度理解——变更是手段，协同是目的。
